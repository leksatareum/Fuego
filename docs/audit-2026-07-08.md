# Fuego Audit - 2026-07-08

## Architecture

Fuego is a Vite + React single-page application for restaurant operations and HACCP follow-up.

The app is currently concentrated in `src/App.jsx`: UI components, business logic, Supabase REST calls, styling, voice commands, printing helpers and navigation all live there. `src/config.js` provides Supabase configuration. `database.sql` initializes the Supabase schema and demo data.

Main modules covered by the UI:

- staff PIN login
- HACCP dashboard and register
- temperature checks
- reception checks
- cooling and reheating tracking
- frying oils
- cleaning
- traceability
- labels and Brother printing flow
- test meals
- recipes and margins
- planning
- setup tasks
- settings
- voice commands

## Changes Applied

- Added `.gitignore` so local env files, dependencies and build output are not committed.
- Added `.env.example` for Vite/Supabase configuration.
- Updated `src/config.js` to read `VITE_SUPABASE_URL` and `VITE_SUPABASE_ANON_KEY`, while keeping the current values as fallback so existing deployments keep working.
- Added `SECURITY.md` documenting the current security posture and hardening path.
- Added `migrations/20260708_runtime_tables.sql` for runtime tables already used by the app: `app_state`, `products`, and `shifts`.
- Added indexes for product lookup and shift lookup.
- Upgraded dev tooling from Vite 5 to Vite 8 and `@vitejs/plugin-react` 6 to remove npm audit vulnerabilities.

## Bugs Fixed

- Existing Supabase installs were missing `app_state`, which is used by the automatic service reset logic.
- Existing Supabase installs were missing `products`, which is used by recipe costing and the product editor.
- Existing Supabase installs were missing `shifts`, which is used by persistent planning.
- Supabase configuration was hardcoded only in source code; the app can now use environment variables.
- npm audit reported one moderate and one high vulnerability through Vite/esbuild; after upgrades, audit reports zero vulnerabilities.

## Verification

Environment used:

- Portable Node.js v22.12.0
- npm install run with `--strict-ssl=false` because this Windows environment rejected the npm registry certificate chain.

Commands run:

```text
npm install --strict-ssl=false
npm audit fix --force --strict-ssl=false
npm install -D @vitejs/plugin-react@latest --force --include=optional --strict-ssl=false
npm run build
npm audit --json --strict-ssl=false
```

Results:

- Production build passed.
- npm audit found 0 vulnerabilities after updates.

## Remaining Risks

- PIN authentication is still client-side. The browser receives user PINs, so this is not secure against a curious or malicious user.
- Admin access is still client-side. Database policies do not enforce role-based admin authorization.
- The broad `allow all` RLS policies are acceptable only for a controlled prototype/internal app. Production should move to authenticated users and restrictive table policies.
- Most write operations do not show detailed database errors to users.
- The very large `src/App.jsx` should eventually be split into modules to reduce regression risk.

## Recommended Next Steps

1. Apply `migrations/20260708_runtime_tables.sql` to existing Supabase projects.
2. Create `.env.local` from `.env.example` for local development/deployment secrets.
3. Move PIN verification to Supabase Auth or an Edge Function.
4. Replace broad RLS policies with role-aware policies.
5. Add Vitest tests for helper functions and Playwright smoke tests for login/navigation.
